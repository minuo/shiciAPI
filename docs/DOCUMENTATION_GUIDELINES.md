# 文档组织规范

## 1. 概述

本文档旨在规范项目文档的组织方式，确保所有新生成的文档文件统一存放在项目根目录下的`docs`目录中，并根据文档类型和功能模块组织到对应的子目录中。遵循此规范将提高文档的可维护性和可访问性。

## 2. 文档存放原则

### 2.1 统一存放要求
- **所有新生成的文档文件必须存放在`docs`目录下**
- **禁止将文档散落在项目的其他目录中**
- **已存在的文档应按规范逐步迁移到正确位置**

### 2.2 目录层次结构
文档应按照类型和功能模块组织到适当的子目录中，遵循以下目录结构：

```
docs/
├── INDEX.md                  # 文档总索引
├── architecture/             # 架构设计文档
├── api/                      # API相关文档
├── development/              # 开发相关文档
├── testing/                  # 测试相关文档
├── deployment/               # 部署相关文档
├── baota/                    # 宝塔面板部署文档
├── configs/                  # 配置相关文档
├── code_reviews/             # 代码审查相关文档
└── security/                 # 安全相关文档
```

## 3. 文档分类标准

### 3.1 按文档类型分类

| 文档类型 | 存放目录 | 包含内容 |
|---------|---------|--------|
| 架构文档 | `docs/architecture/` | 系统架构、数据库设计、技术选型 |
| API文档 | `docs/api/` | API规范、接口说明、Swagger文档 |
| 开发文档 | `docs/development/` | 编码规范、开发指南、工作流程 |
| 测试文档 | `docs/testing/` | 测试计划、测试用例、测试报告 |
| 部署文档 | `docs/deployment/` | 部署步骤、环境配置、监控维护 |
| 宝塔文档 | `docs/baota/` | 宝塔面板特定的部署指南和配置 |
| 配置文档 | `docs/configs/` | 系统配置、环境变量、性能调优 |
| 代码审查 | `docs/code_reviews/` | 代码审查报告、质量分析、重构建议 |
| 安全文档 | `docs/security/` | 安全指南、风险评估、安全策略 |

### 3.2 按功能模块分类

对于特定功能模块的文档，可以在对应类型目录下创建子目录，例如：

```
docs/api/
├── user_api/
│   └── user_authentication.md
└── poem_api/
    └── poem_management.md
```

## 4. 文档命名规范

### 4.1 通用命名规则

- **使用英文命名**，避免使用中文文件名
- **文件名应简洁明了**，能准确反映文档内容
- **使用下划线(_)分隔单词**，避免空格和特殊字符
- **重要文档使用全大写命名**，普通文档使用驼峰命名或小写加下划线

### 4.2 文档类型命名格式

| 文档类型 | 命名格式 | 示例 |
|---------|---------|------|
| 指南类 | `XXX_GUIDE.md` | `DEPLOYMENT_GUIDE.md` |
| 报告类 | `XXX_REPORT.md` | `API_TEST_REPORT.md` |
| 规范类 | `XXX_STANDARDS.md` | `CODING_STANDARDS.md` |
| 设计类 | `XXX_DESIGN.md` | `DATABASE_DESIGN.md` |
| 配置类 | `XXX_CONFIG.md` | `ENVIRONMENT_CONFIG.md` |

## 5. 提交代码时的文档检查

### 5.1 文档存放检查要求

在提交代码时，开发者必须检查以下内容：

- ✅ 确保所有新增文档都存放在`docs`目录下
- ✅ 确保文档已放入正确的子目录中
- ✅ 确保文档命名符合规范
- ✅ 确保已在`INDEX.md`中更新文档索引

### 5.2 提交前检查清单

提交代码前，请执行以下检查：

1. **检查文档位置**：`find . -path './docs' -prune -o -name '*.md' -print | grep -v './node_modules' -v './docs'`
   - 如果有输出结果，表示有Markdown文档不在docs目录中

2. **检查文档命名**：确保文件名符合命名规范，无中文和特殊字符

3. **更新索引**：确认已在`INDEX.md`中添加新文档条目

## 6. 文档格式要求

### 6.1 Markdown格式规范

- 使用标准Markdown语法
- 包含适当的标题层级（#、##、###等）
- 使用表格展示结构化数据
- 使用代码块展示代码示例
- 添加适当的图片和图表

### 6.2 文档内容要求

- **标题页**：包含文档标题、版本、创建日期和作者
- **概述**：简要描述文档目的和范围
- **目录**：对于长文档，添加目录便于导航
- **正文**：清晰组织的内容，使用适当的格式
- **参考资料**：相关文档和资源的链接
- **变更日志**：记录文档的重要更新

## 7. 文档维护流程

### 7.1 更新和维护

- 代码修改时，同步更新相关文档
- 功能变更时，更新对应的设计和API文档
- 定期审查文档的准确性和完整性

### 7.2 版本控制

- 文档应与代码一起进行版本控制
- 重要文档应包含版本号和变更历史
- 文档更新应在代码提交信息中注明

## 8. 最佳实践

### 8.1 文档创建

- 创建新功能时，同步创建相应的文档
- 使用模板快速生成标准化文档
- 保持文档简洁明了，重点突出

### 8.2 文档使用

- 团队成员应熟悉文档目录结构
- 优先查阅最新的文档
- 发现文档问题时及时反馈和更新

## 9. 违规处理

对于不遵循文档组织规范的提交，团队可以：

- 要求开发者在合并前修正文档位置
- 记录文档违规情况，纳入代码审查标准
- 定期整理和规范化文档存放

---

**注意**：本规范应在团队内部广泛宣传和执行，确保所有成员理解并遵循文档组织标准，共同维护项目文档的整洁和高效。
